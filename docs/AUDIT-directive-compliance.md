# AUDIT: Pv2 Compliance with Agent-First Directive

**Date:** 2026-04-20  
**Auditor:** Schellie  

---

## 1. GAPS

### GAP-1: UI Makes Direct Database Writes (CRITICAL)
**Current:** Every action (assign, promote, demote, mark read, save contact, send reply) goes directly from React component → Supabase client → database.  
**Required:** UI → MCP Tool → Database  
**Impact:** Agents cannot execute any action the UI does. Complete architectural violation.  
**Fix:** Build MCP tool layer for ALL actions, refactor UI to call MCP tools.

### GAP-2: No MCP Tool Layer Exists
**Current:** Zero MCP tools for CRM actions. The pv2-mcp server only has read-only tools (search_communities, get_floor_plans, etc.) for the Schellie chatbot.  
**Required:** Full CRUD MCP tool set: assign_opportunity(), promote_prospect(), send_message(), create_contact(), update_contact(), log_activity(), etc.  
**Impact:** Agents have zero ability to take any CRM action.  
**Fix:** Design and build CRM MCP tool layer.

### GAP-3: No Shadow Agents
**Current:** AI "suggestions" are hardcoded template strings, not generated by agents. No agent evaluates queue items. No confidence scores. No reasoning.  
**Required:** Shadow agents running on all workflows generating real recommendations.  
**Impact:** No agent decision data being collected. No validation of agent performance. No path to Auto Mode.  
**Fix:** Implement shadow agent framework that evaluates each queue item, prospect, and communication.

### GAP-4: No Action Attribution
**Current:** Stage transitions log `triggered_by: "manual"` but no confidence_score, no reasoning, no agent_name. Activities log channel but not who decided to send.  
**Required:** Every action: triggered_by, agent_name, confidence_score, reasoning, timestamp.  
**Impact:** Cannot audit agent vs human decisions. Cannot measure agent accuracy. Cannot transition to Auto Mode safely.  
**Fix:** Extend stage_transitions and activities tables with attribution fields. Add attribution to every MCP tool call.

### GAP-5: No System Mode Support
**Current:** System only operates in "Manual Mode" — human does everything. No concept of Assisted or Auto mode.  
**Required:** Mode toggle: Manual / Assisted / Auto per workflow.  
**Impact:** Switching to Pv3 requires development, not configuration.  
**Fix:** Add system_mode field to workflow configurations. MCP tools check mode before executing.

### GAP-6: Business Logic in UI Components
**Current:** Queue bucketing logic, contact matching, template generation, assign lane suggestion — all in React components or inline in page files.  
**Required:** All business logic in MCP tools or server-side functions.  
**Impact:** Agents cannot access the same logic. Duplication risk. UI changes can break business rules.  
**Fix:** Extract all business logic into MCP tools.

## 2. RISKS FOR PV3

### RISK-1: Complete Rewrite Required (HIGH)
Every UI action currently bypasses any agent-callable layer. Moving to Pv3 (agents execute) would require rewriting every component to call MCP tools instead of Supabase directly. This is the exact scenario the directive says must be avoided.

### RISK-2: No Agent Training Data (MEDIUM)
Without shadow agents running, there is no data about what agents WOULD have recommended. When Pv3 arrives, agents start with zero calibration data. Shadow mode should be collecting data NOW.

### RISK-3: Template Suggestions Won't Scale (MEDIUM)
Current "AI suggestions" are `if/else` strings based on form type. Real AI recommendations need the scoring engine, contact profiles, and behavioral signals. The current approach gives users false expectations of "AI" that's actually hardcoded logic.

### RISK-4: No Confidence Calibration (MEDIUM)
Without confidence scores being logged alongside human decisions, there's no way to measure agent accuracy or set appropriate thresholds for Auto Mode.

## 3. REQUIRED CHANGES (Prioritized)

### P0 — MCP Tool Layer (MUST DO FIRST)
Build the CRM MCP tool set that both UI and agents call:

```
QUEUE TOOLS:
  evaluate_queue_item(opp_id) → { recommendation, confidence, reasoning }
  assign_opportunity(opp_id, stage, community_id, reason) → { success, transition_id }
  
COMMUNICATION TOOLS:
  send_email(contact_id, subject, body, from_user_id) → { activity_id }
  send_sms(contact_id, body, from_user_id) → { activity_id }
  generate_response(opp_id, channel, form_type) → { subject, body, confidence }

PIPELINE TOOLS:
  promote_opportunity(opp_id, new_stage, reason) → { success, transition_id }
  demote_opportunity(opp_id, new_stage, reason) → { success, transition_id }
  
CONTACT TOOLS:
  create_contact(data) → { contact_id }
  update_contact(contact_id, data) → { success }
  match_contact(email, phone, name) → { contact_id, match_type }
  
ACTIVITY TOOLS:
  log_activity(contact_id, channel, direction, subject, body) → { activity_id }
  mark_read(activity_id) → { success }
```

Every tool logs: action_type, triggered_by, agent_name, confidence_score, reasoning, timestamp.

### P1 — Refactor UI to Call MCP Tools
Replace all direct Supabase calls in UI with MCP tool calls:
- OscClient.tsx: handleAction() → call assign_opportunity()
- CommunityDashboard.tsx: promote/demote → call promote_opportunity()
- CommHub.tsx: mark read, send reply → call mark_read(), send_email()
- OpportunityPanel.tsx: save contact → call update_contact()

### P2 — Shadow Agent Framework
Build the shadow agent that:
- Evaluates each queue item when it arrives
- Generates a recommendation (lane + confidence + reasoning)
- Stores the recommendation in the opportunity record
- UI displays: "Recommended: Lead · Cardinal Grove (92%) — Reason: schedule_visit + community interest"
- Human approves or overrides
- Both decisions are logged for comparison

### P3 — Action Attribution Schema
Extend tables:
```sql
ALTER TABLE stage_transitions ADD COLUMN confidence_score numeric(5,2);
-- reasoning already exists
ALTER TABLE stage_transitions ADD COLUMN agent_name text;

ALTER TABLE activities ADD COLUMN triggered_by text DEFAULT 'human';
ALTER TABLE activities ADD COLUMN agent_name text;
ALTER TABLE activities ADD COLUMN confidence_score numeric(5,2);
ALTER TABLE activities ADD COLUMN reasoning text;
```

### P4 — System Mode Configuration
```sql
CREATE TABLE system_config (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  key text UNIQUE NOT NULL,
  value jsonb NOT NULL,
  updated_at timestamptz DEFAULT now()
);

INSERT INTO system_config (key, value) VALUES
  ('queue.mode', '"assisted"'),        -- manual | assisted | auto
  ('prospect.mode', '"assisted"'),
  ('communication.mode', '"manual"'),
  ('scoring.mode', '"assisted"');
```

MCP tools check mode:
- Manual: execute immediately (no recommendation)
- Assisted: generate recommendation, wait for approval
- Auto: generate recommendation, execute immediately
